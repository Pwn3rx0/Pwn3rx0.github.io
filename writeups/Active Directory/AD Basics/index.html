What's the AD?

Active Directory (AD) is a centralized database and service developed by Microsoft to manage users, computers, and resources in a Windows network. It provides authentication, authorization, and policy enforcement across all devices in a domain, making it the core of enterprise identity management.

Why do we use Active Directory?

  • To centralize identity management, so administrators can create, manage, and secure user accounts and computers from one place.
  • To enforce security policies across the network (e.g., password rules, software restrictions, or access permissions).
  • To improve scalability, allowing organizations with hundreds or thousands of computers and users to manage them efficiently.
  • To provide seamless access, so users can log in with the same credentials on any domain-joined machine.

How does Active Directory work?

  • Domain & Domain Controller: All resources (users, computers, groups) are stored in a central database on a server called the Domain Controller (DC).
  • Authentication: When a user logs into a computer, the credentials are sent to the DC. The DC checks them against AD using authentication protocols (Kerberos or NTLM).
  • Authorization: Once authenticated, AD determines what the user can access (files, printers, applications) based on permissions and group memberships.
  • Policy Enforcement: Group Policies (GPOs) are applied to users and computers to enforce security settings and configurations across the network.
  • Scalability: AD can connect multiple domains into Trees and Forests, allowing large organizations to manage thousands of users and devices with trust relationships.

Organizational Units (OUs)

 are container objects that allow you to classify users and machines. OUs are mainly used to define sets of users with similar policing requirements.

  • Keep in mind that a user can only be a part of a single OU at a time.
  • Default Containers created by Windows include:
    • Builtin : Contains default groups for all Windows hosts.
    • Computers : Default location for newly joined machines.
    • Users : Contains default domain-wide user and group objects.
    • Domain Controllers : Holds the DC accounts.
    • Managed Service Accounts : Holds service-specific accounts.

Security Groups vs. OUs

Both groups and OUs are used for organizing users and computers, but they serve different purposes:

  • Organizational Units (OUs):
    • Used for policy application (security settings, software deployment, restrictions).
    • A user can only belong to one OU.
  • Security Groups:
    • Used for permission management (files, folders, printers, shared drives).
    • A user can belong to multiple groups.

While there is no golden rule on how to organise your machines, an excellent starting point is segregating devices according to their use. In general, you'd expect to see devices divided into at least the three following categories:

  • 1. Workstations* Workstations are one of the most common devices within an Active Directory domain. Each user in the domain will likely be logging into a workstation. This is the device they will use to do their work or normal browsing activities. These devices should never have a privileged user signed into them.

  • 2. Servers* Servers are the second most common device within an Active Directory domain. Servers are generally used to provide services to users or other servers.

  • 3. Domain Controllers* Domain Controllers are the third most common device within an Active Directory domain. Domain Controllers allow you to manage the Active Directory Domain. These devices are often deemed the most sensitive devices within the network as they contain hashed passwords for all user accounts within the environment.

Group Policy Objects (GPO)

GPOs are simply a collection of settings that can be applied to OUs. GPOs can contain policies aimed at either users or computers, allowing you to set a baseline on specific machines and identities.

What is a GPO

  • A Group Policy Object (GPO) is a collection of settings that can apply to either users or computers.
  • GPOs are created in the Group Policy Management Console (GPMC) and then linked to OUs.
  • Policies apply to the OU and all of its sub-OUs.
  • By default, GPOs apply to Authenticated Users (all users and computers in the OU). Security Filtering can be used to narrow the scope.

Viewing & Editing GPOs

  • Open Group Policy Management from the Start Menu.
  • Under Group Policy Objects, create a new GPO or select an existing one (e.g., Default Domain Policy).
  • Scope tab → shows where the GPO is linked.
  • Settings tab → lists all active policies inside the GPO (divided into Computer Configuration and User Configuration).
  • Edit GPO → navigate through:
    • Computer Configuration → Policies → Windows Settings → Security Settings_
    • User Configuration → Policies → Administrative Templates

When using Windows domains, all credentials are stored in the Domain Controllers. Whenever a user tries to authenticate to a service using domain credentials, the service will need to ask the Domain Controller to verify if they are correct. Two protocols can be used for network authentication in windows domains:

  • Kerberos: Used by any recent version of Windows. This is the default protocol in any recent domain.
  • NetNTLM: Legacy authentication protocol kept for compatibility purposes.

Kerberos Authentication

Authentication based on tickets instead of resending credentials.

  • Steps:*
  1. Initial Authentication (AS-REQ / AS-REP):
    • User → Authentication Service (AS) on the Key Distribution Center KDC (on DC).
    • Sends: username + timestamp, encrypted with key derived from user password.
    • KDC responds with:
      • Session Key – used for further communication.
      • TGT encrypted with krbtgt account hash (user cannot read).
  2. Request Service Ticket (TGS-REQ / TGS-REP):
    • User wants a service (e.g., file share, DB).
    • Sends to KDC: TGT + username + timestamp (encrypted with Session Key) + SPN (service identifier).
    • KDC responds with:
      • TGS (Ticket Granting Service) – specific to service.
      • Service Session Key.
    • TGS encrypted with service account hash (service can decrypt).
  3. Service Authentication:
    • User → sends TGS to service.
    • Service decrypts TGS using its account hash → validates Service Session Key.
    • Connection established.
  • Notes:*
  • TGT allows requesting service tickets without re-entering password.
  • KDC doesn’t store Session Key → can regenerate by decrypting TGT.

NetNTLM Authentication

  • Challenge-response* mechanism (no password sent across network).
  • Steps:*
  1. Client → sends authentication request to server.
  2. Server → sends random challenge.
  3. Client → uses NTLM password hash + challenge + known data → generates response, sends back.
  4. Server → forwards challenge + response to DC.
  5. DC → recalculates response using stored hash → compares with client’s response.
    • Match = authentication successful.
  6. Server → relays result back to client.
  • Notes:*
  • Password (or hash) never transmitted directly.
  • Domain account: DC verifies challenge-response.
  • Local account: Server verifies itself using local SAM (has password hash).

1. Single Domain

  • One domain (e.g., thm.local) can manage users, computers, and servers.
  • Scales well at first but becomes harder as companies expand and diversify.

2. Trees

  • Definition: Multiple domains in the same namespace grouped together.
  • Example:
    • Root domain: thm.local.
    • Subdomains: uk.thm.local, us.thm.local.
  • Benefits:
    • Partitioned management (UK IT manages UK, US IT manages US).
    • Independent Domain Controllers per branch.
    • Policies configured separately.
  • Admin roles:
    • Domain Admins → control their specific domain.
    • Enterprise Admins → admin privileges across all domains in enterprise (tree/forest).

3. Forests

  • Definition: Union of multiple trees with different namespaces.
  • Example:
    • Company THM (thm.local).
    • Company MHT (mht.local).
    • Together → form a forest.
  • Purpose: Allows separate organizations/domains to coexist in the same network environment.
  • Each tree keeps its independent namespace but can interact via trusts.

Trust Relationships

  • Purpose: Allow users in one domain to access resources in another.
  • Example: THM UK user accessing MHT EU server.
  • Types:
    • One-way trust: Domain A → trusts Domain B → B users can access A’s resources (not vice versa).
    • Two-way trust: Both domains mutually trust each other → both sides can access cross domain resources.
    • Default: Two-way trust automatically created when domains join a tree/forest.
  • Important: Trusts only enable the possibility of cross-domain access.
    • Actual access requires explicit authorization to resources.
